Objective
Build a full-stack decentralized application featuring a custom oracle system. You will create an off-chain Node.js service to fetch real-world sports data, a suite of Solidity smart contracts to manage data and betting markets, and a React frontend for user interaction. You'll learn to solve the blockchain oracle problem, integrate external APIs with smart contracts securely, and build event-driven financial applications on the blockchain. This project is vital for understanding how real-world data powers decentralized finance (DeFi) and other on-chain applications.

Description
Background
In the blockchain ecosystem, smart contracts are powerful, self-executing programs. However, they operate in a deterministic, isolated environment and cannot natively access data from the outside world, such as sports scores, weather information, or financial asset prices. This limitation is known as the "Oracle Problem." Oracles are services that bridge this gap by fetching external data, verifying it, and delivering it onto the blockchain for smart contracts to use.

This project simulates a common use case for oracles: a sports betting platform. You will build a complete system consisting of three main components: an off-chain oracle service, on-chain smart contracts, and a web-based frontend. The oracle service will run on a traditional server (locally in this case), pull player statistics from a public sports API, and submit this data to your smart contract. The smart contract acts as the on-chain source of truth, holding the data and the logic for betting markets. Users will interact with a React DApp to place bets, and the system will automatically settle them once the oracle finalizes the match results. This architecture demonstrates a production-ready pattern for building hybrid smart contract applications that combine the power of off-chain computation with the security and trustlessness of the blockchain.

This project is divided into three main parts: the blockchain smart contracts, the off-chain oracle service, and the frontend web application. You will use a containerized setup with Docker to manage these services.

Implementation Guidelines
These are advisory guidelines to help you build a more robust and production-ready system.

Security: The oracle's private key is a critical secret. While using a .env file is acceptable for this project, in a production environment, you should use a dedicated secrets management service (like AWS Secrets Manager or HashiCorp Vault) to protect it. Never commit your .env file to version control.

Error Handling: Your oracle service must be resilient to API failures. Implement retry logic with exponential backoff for API requests. Log all errors and transaction failures for easier debugging. Consider what should happen if a sports API returns invalid or unexpected data.

Data Integrity: For this project, a single oracle with an Ownable contract is sufficient. In a real-world DeFi protocol, you would want to avoid a single point of failure. Consider how you might extend this system to use multiple oracles and aggregate their data on-chain to increase trust and reliability.

Gas Optimization: Smart contract transactions cost gas. When the oracle submits data, it should be as efficient as possible. Structure your data and functions to minimize storage writes, which are the most expensive operations. For example, packing multiple data points into a single struct can be more efficient than storing them as individual variables.

Event-Driven Frontend: Your frontend should listen for smart contract events (e.g., DataFinalized, BetSettled). This allows the UI to update in real-time without requiring users to manually refresh the page, providing a much better user experience.

FAQ
Q: Which sports API should I use?

A: We recommend starting with TheSportsDB API, as it's free and does not require an API key. This simplifies setup and allows you to focus on the core logic. You are free to use others, but be mindful of rate limits on free tiers.
Q: How can the betting contract trust the oracle contract?

A: The trust is established at deployment. The BettingMarket contract's constructor is given the address of the SportsOracle contract (see requirement contract-deployment). It will only read data from that specific address.
Q: What happens if the oracle submits wrong data?

A: In this single-oracle model, that is the primary trust assumption. The system is only as trustworthy as the oracle operator. This is why decentralized oracle networks (like Chainlink) are popular in production. Your design should acknowledge this limitation.
Q: How do I test the oracle's interaction with the smart contract without waiting for a real game?

A: Your oracle service must expose manual trigger endpoints (see oracle-service-api-trigger-update and oracle-service-api-trigger-finalize). This allows you to simulate a game's conclusion by pushing data on-demand during testing.
Q: Do I need to deploy to a public testnet?

A: While final deployment to a testnet like Sepolia is a great way to showcase your work, all core requirements can be fulfilled and verified using a local Hardhat network. The docker-compose.yml setup is designed for local evaluation.
Q: How does the oracle prove the data is authentic?

A: In this architecture, authenticity is enforced by access control via the onlyOracle modifier on the SportsOracle contract (contract-oracle-access-control). Only the address that deployed the contract (or the one it designates) can write data. In more advanced systems, the oracle would cryptographically sign the data off-chain, and the smart contract would verify the signature on-chain.

4.
The SportsOracle contract must have a function to allow the oracle to submit player performance data.

Contract Specification:

Function Signature: function submitPlayerData(uint256 matchId, uint256 playerId, uint256 pointsScored) external
Behavior:
Must only be callable by the oracleAddress (enforced by onlyOracle modifier).
Must not be callable if the performance data for that matchId and playerId is already finalized.
Must update the performances mapping with the provided data.
Must emit a DataSubmitted event upon successful execution.
Verification:

A transaction will be sent to submitPlayerData from the designated oracle address. The DataSubmitted event will be checked, and the state change in the performances mapping will be verified by a contract call.
5.
The SportsOracle contract must have a function to allow the oracle to finalize match data, preventing further updates.

Contract Specification:

Function Signature: function finalizeMatch(uint256 matchId, uint256 playerId) external
Behavior:
Must only be callable by the oracleAddress.
Must set the finalized flag to true for the specified matchId and playerId in the performances mapping.
Must emit a DataFinalized event.
Verification:

After submitting data, a transaction will be sent to finalizeMatch. The DataFinalized event will be checked, and the finalized flag will be verified via a contract call.

Contract Specification:

Functions to Protect: submitPlayerData, finalizeMatch

Verification:

An attempt will be made to call submitPlayerData from an account that is NOT the oracleAddress.
The transaction must revert with an appropriate error message (e.g., "Only oracle can call").

9.
The BettingMarket contract must allow for the settlement of a bet after a match is finalized, paying out to winners.

Contract Specification:

Function Signature: function settleBet(uint256 betId) external
Behavior:
Must revert if the bet is already settled.
Must revert if the match is not yet finalized in the oracle contract.
Must fetch the actual player performance from the oracle contract.
Must determine if the bet was won. For an over/under bet, this means actualPoints > predictedValue.
If the bet was won, it must transfer a payout (e.g., 2x the original amount) to the bettor.
Must mark the bet as settled.
Verification:

A match will be finalized via the oracle. A call to settleBet will be made for a winning bet.
The bettor's ETH balance change will be checked to confirm the payout was received. The bet's settled status will be verified.
